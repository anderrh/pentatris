Pentatris — Adding Pentominoes to Your Tetris

Tutorial 2: Extending 7 tetrominoes to 25 pieces (7 + 18 pentominoes)

Prerequisite: Complete Tutorial 1 (asm_tutorial.html) first. You should have a fully working Tetris game with all 35 tests passing.

What Are Pentominoes?

Pentominoes are shapes made of 5 connected squares, just as tetrominoes are made of 4. There are 18 one-sided pentominoes (counting mirror images as distinct). Combined with the 7 standard tetrominoes, your game will have 25 different pieces.

Each pentomino fits in a 40×40 pixel frame (5×5 tiles) and has 4 rotation states, just like tetrominoes.

Overview of Changes

The beauty of this upgrade is that no game logic code needs to change. The collision detection, row clearing, input handling, and rendering all work with any number of pieces and any number of tiles per piece. You only need to:

  1. Regenerate graphics data — include pentomino PNGs in the build pipeline
  2. Fix a sprite cleanup bug — pentominoes have 5 sprites in their preview, but tetrominoes only have 4
  3. Add tests — verify pentominoes work with existing game logic

Step 1: Enable Pentomino Graphics

1a. Add Pentomino PNG Files

You should already have 18 pentomino PNG files in your Graphics/ directory, named Pentomino1.png through Pentomino18.png. Each is a 160×40 sprite sheet (4 rotation states, each 40×40 pixels).

1b. Update the Makefile

The Makefile already has pentomino support gated behind the PENTO flag. Change the default from 0 to 1:

File: Makefile

PENTO ?= 0
PENTO ?= 1

This single change tells the build system to:

  • Include Graphics/Pentomino*.png in the sprite list
  • Run png2asset on each pentomino PNG (40×40 frames instead of 32×32)
  • Run tools/png2asm.py to regenerate all assembly include files

1c. Regenerate Graphics

make clean
make gfx

This regenerates four key files:

FileWhat Changed
gfx/sprite_defs.incAdded 18 PentominoN_TILE_COUNT constants, PIECE_COUNT = 25, TOTAL_SPRITE_TILES includes all 25 pieces
gfx/sprite_data.incAdded pentomino tile bytes, metasprite definitions, extended pointer table to 100 entries (25×4), extended tile offset table to 25 entries
gfx/setup_sprites.incAdded set_sprite_data and set_bkg_data calls for each pentomino
gfx/palette_data.incUnchanged — pentominoes reuse the same 8 CGB palettes
Key insight: Because the metasprite pointer table, tile offset table, and PIECE_COUNT are all auto-generated, every function that uses them (_CanPieceBePlacedHere, _PickNewTetromino, _SetCurrentPieceInBackground, _SetupVRAM, etc.) automatically works with pentominoes. No code changes needed!

Step 2: Fix the Preview Sprite Cleanup

There is one subtle bug when mixing pentominoes and tetrominoes. The next-piece preview in _UpdateGui uses move_metasprite starting at sprite slot 16. A tetromino has 4 sprites (slots 16–19), but a pentomino has 5 (slots 16–20).

When a pentomino preview is replaced by a tetromino preview, sprite slot 20 retains the old pentomino sprite, creating a visual ghost. The fix is to hide sprite 20 before drawing the new preview.

File: asm/hud.asm, inside _UpdateGui

Add this block after the _DrawNumber calls and before the move_metasprite preview code:

    ; Hide the 5th preview sprite slot (index 20) which is stale when a
    ; pentomino preview (5 cells) is replaced by a tetromino (4 cells).
    ; move_metasprite overwrites slots 16-19 (or 16-20), so only 20 needs clearing.
    ld a, 20
    ld b, 0
    ld c, 0
    call move_sprite

This moves sprite 20 to position (0, 0), which is offscreen (the Game Boy's OAM coordinate system places (0, 0) at 8 pixels left and 16 pixels above the visible area).

How It All Works

The elegance of the data-driven design becomes clear here. Every game function operates on the same abstractions:

  • Metasprite tables: Each piece (tetromino or pentomino) is just an index into _Tetrominos_metasprites. The table grew from 28 to 100 entries, but the lookup code is identical.
  • Tile offsets: _tileOffsets grew from 8 to 26 entries. Each piece's base tile is still just _tileOffsets[piece_index].
  • PIECE_COUNT: _RandomNumber(0, PIECE_COUNT) and _PickNewTetromino automatically distribute across all 25 pieces because PIECE_COUNT is defined in the auto-generated sprite_defs.inc.
  • Sentinel-based iteration: _CanPieceBePlacedHere and _SetCurrentPieceInBackground loop until the $80 sentinel. They don't care if a piece has 4 tiles or 5 — they just keep reading until the end.

The only code change was the sprite cleanup fix, which is a rendering quirk rather than a logic issue. This demonstrates the value of designing data-driven game engines: adding content requires zero logic changes.

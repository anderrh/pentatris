Game Boy Tetris — RGBDS Assembly Tutorial

Learn Game Boy assembly by building Tetris from the ground up

Introduction

Tetris is an memorable and fun retro game. Players rotate and place blocks of falling shapes made of four connected squares — to fill complete rows across a 10-column playing field. Completed rows are cleared from the board, earning points. If the stack of pieces reaches the top, the game is over.

In this tutorial, inspired by Larold's C tetris totrial you will re-create Tetris for the Nintendo Game Boy, written entirely in RGBDS assembly. The core gameplay is fundamentally a series of array operations: checking if a piece fits, copying tiles to the background, scanning rows, and shifting tiles down. The game framework — hardware initialization, the main loop, graphics data, and runtime library — is already provided. Your job is to fill in the 18 functions that make the game work.

Each section describes one function: what it does, which registers carry its inputs and outputs, and hints about the algorithm. When you get stuck, click the spoiler card to reveal the reference implementation.

Prerequisites: RGBDS v1.0.0 installed, basic understanding of 8-bit registers (A, B, C, D, E, H, L), the stack (push/pop), and Game Boy memory layout. The inc/runtime.inc file documents every runtime function you can call.

Each tetromino rotation is stored as a metasprite — a list of 8×8 sprite entries, each containing {dy, dx, dtile, props}, terminated by a sentinel byte $80. The metasprite pointer table and tile offset table are auto-generated in gfx/sprite_data.inc:

The Main Loop: Code You Get for Free (and Why It Matters)

The main game loop lives in asm/main.asm and is already written for you. But understanding it is essential — it's the conductor that calls every function you'll implement. Here's the full loop, annotated so you can see exactly where your code fits in.

Startup Sequence

Before the loop begins, main.asm calls your setup functions with the LCD turned off (so VRAM writes are safe), then turns the display on and initializes the music:

_main::
    di                              ; disable interrupts during setup
    call _SetupVRAM                 ; YOUR CODE: load all tile data into VRAM
    call _SetupUserInterface        ; YOUR CODE: draw the playing field
    DISPLAY_ON                      ; turn the screen on

    ; Enable audio hardware and start background music
    ld a, $80
    ldh [rNR52], a                  ; sound on
    ld a, $FF
    ldh [rNR51], a                  ; all channels to both speakers
    ld a, $77
    ldh [rNR50], a                  ; max volume
    ld hl, _tetris_music
    call hUGE_init                  ; initialize the music driver
    ei                              ; enable interrupts (music plays via VBlank)

The Game Loop, Step by Step

Every frame (60 times per second), the loop runs through these phases. Functions marked "YOUR CODE" are ones you implement. The rest is provided.

.GameplayStart:
    call _SetupGameplay             ; YOUR CODE: clear board, pick first piece, zero score

.mainLoop:
    ; --- PHASE 1: Do we need a new piece? ---
    ; When a piece locks, _currentTetromino is set to 255 (meaning "no piece").
    ; The loop detects this and tries to spawn the next one.
    ld a, [_currentTetromino]
    cp 255                          ; 255 = no active piece
    jr nz, .hasTetromino            ; if we have a piece, skip spawning

    call _PickNewTetromino          ; YOUR CODE: try to spawn the "next" piece
    or a                            ; did it return 0?
    jr z, .GameplayStart            ; yes -> board full! restart the game (game over)
    call _UpdateGui                 ; YOUR CODE: redraw HUD with new "next" preview

.hasTetromino:
    ; --- PHASE 2: Animate and handle input ---
    call _AnimateBackground         ; YOUR CODE: advance border tile animation
    call _HandleInput               ; YOUR CODE: read joypad, move/rotate piece

    ; Draw the active piece on screen as a floating metasprite.
    ; This code computes: metasprite = Tetrominos_metasprites[piece*4 + rotation]
    ; then calls move_metasprite(metasprite, tileOffsets[piece], 0, currentX*8, currentY*8)
    ; ... (about 20 lines of table lookup and coordinate math, provided for you)

    ; --- PHASE 3: Gravity and piece locking ---
    ; YOUR CODE: This is the heartbeat of the game. Every frame, the fall
    ; timer ticks up. At 30, try to move the piece down. If it can't move,
    ; start a lock delay. When the lock expires, lock the piece: score it,
    ; copy it to the background, blink/clear full rows, hide sprites.
    call _UpdateFallTimer           ; YOUR CODE: gravity + lock sequence

.noLock:
    call wait_vbl_done              ; YOUR CODE: wait for screen refresh
    call run_dma                    ; copy shadow OAM to hardware (provided for you)
    jp .mainLoop                    ; repeat forever

Two Ways to Draw Tetrominos

Notice the two rendering paths in the loop above:

  1. The active piece (sprites): Every frame, move_metasprite positions hardware sprites on screen. Sprites float above the background and can be repositioned freely. This is Phase 2 of the loop.
  2. Locked pieces (background tiles): When the lock delay expires (Phase 4), _SetCurrentPieceInBackground copies the piece's tiles into the background tilemap. Once written to the background, they become part of the static board and are subject to row clearing.

The User Interface

The background tilemap is loaded from a pre-made PNG image. The play area occupies columns 2–11 (10 tiles wide), rows 0–17 (18 tiles tall). The HUD area is to the right, showing score, level, lines, and the next piece preview.

The Green and Beige Tiles

Two special tiles in the UserInterface graphic have important roles:

  • Green tiles (tile at column 0, row 0): These are the animation base tiles. During setup, every instance of this tile is replaced with one of two alternating animation tile slots (TILEANIMATION_TILE1_VRAM / TILEANIMATION_TILE2_VRAM) in a checkerboard pattern. The animation system then swaps the VRAM data in these slots each frame, creating the animated border effect.
  • Beige tiles (tile at column 3, row 3): These are the blank tiles. The play area is filled with this tile when empty. _CanPieceBePlacedHere checks whether a cell contains this tile to determine if it's free. Row clearing functions use it to detect which rows are full (all non-blank).

Both tiles are read from VRAM after the tilemap is loaded (in _SetupUserInterface) and stored in global variables _tileAnimationBase and _blankTile. If you change these tiles in the graphic, the game logic will break.

The Window Layer Trick

The Game Boy has two tilemap layers: background and window. This game only uses the background for display, but the window layer's memory is repurposed as a temporary buffer during the blink animation.

When _BlinkFullRows detects a full row, it copies that row's tiles from the background to the window layer (both tile data and palette attributes). During the blink animation, it alternates between blanking the row (filling with _blankTile) and restoring it from the window layer copy. Without this trick, once the row is blanked, the original tile data would be lost. After the animation completes, the window layer is cleared.

Music

Background music is handled by hUGEDriver, a Game Boy sound driver. The music data is in asm/tetris_music.asm (auto-converted from a .mod tracker file). The driver is initialized in main.asm and its hUGE_dosound function is called every VBlank interrupt by the header's ISR. You don't need to implement anything for music — it's all boilerplate in asm/header.asm and asm/main.asm.

How 18 Functions Make a Game of Tetris

A game of Tetris is surprisingly simple at its core. Pieces fall, the player moves them, they lock in place, full rows disappear, and the score goes up. But translating that into code requires solving a handful of distinct problems. Here is how the 18 functions you will implement fit together to make the whole thing work.

Foundation: The Building Blocks

Before tackling any game logic, you'll implement four tiny runtime functions that nearly every other function calls. These teach the fundamental patterns of Game Boy programming: computing memory addresses from indices, reading and writing VRAM, and saving registers across function calls.

  • move_sprite Easy — Positions one sprite on screen by writing its X and Y coordinates into the shadow OAM buffer. You'll compute the buffer address from the sprite index (index × 4), which is the same address-from-index pattern used everywhere in the game. Only 8 instructions.
  • set_sprite_tile Easy — Assigns a tile graphic to a sprite. Almost identical to move_sprite but writes to a different byte offset. Reinforces the OAM layout: [Y, X, tile, attributes].
  • get_bkg_tile_xy Easy — Reads one tile from the background tilemap. Just 3 instructions — calls a helper to compute the VRAM address, waits for VRAM access, and reads the byte. This is how the game checks whether a board cell is empty or occupied.
  • set_bkg_tile_xy Easy — Writes one tile to the background. Slightly trickier because you must save the tile value (in D) before the helper call clobbers it — your first encounter with the push/pop pattern that you'll use constantly throughout the rest of the tutorial.

Startup: Getting Pixels on Screen

Before anything can happen, the Game Boy's video memory needs to be loaded with tile graphics and the screen needs to show the playing field. Three functions handle this:

  • _SetupVRAM Easy — Copies all the raw pixel data into the Game Boy's video memory: the tetromino sprites, the background tiles for the playing field border, the number font for the score display, and the color palettes. Without this, the screen is blank.
  • _SetupUserInterface Hard — Draws the playing field border, score labels, and decorative tiles onto the background tilemap. It also detects which tile is "blank" (the empty play area) and which tile is the animated border, reading them directly from the loaded graphic so the game logic adapts automatically to the art.
  • _SetupAnimatedBackground Medium — Replaces every border tile with a checkerboard of two animation slots, so that the border can shimmer during gameplay. A small visual touch that makes the game feel alive.

Each Frame: Keeping Things Moving

Every frame (60 times per second), the game needs to animate the border and respond to button presses:

  • _AnimateBackground Medium — Advances a frame counter and swaps the tile data in the two animation VRAM slots. Because every border tile on screen points to one of these two slots, changing the data in the slot changes the appearance of every border tile at once — an efficient trick for animating many tiles without touching the tilemap.
  • _HandleInput Hard — Reads the joypad and translates button presses into game actions. Down accelerates the fall. The A button rotates the piece, trying three positions (center, one right, one left) to implement basic "wall kicks" so rotation works even against walls. Left and right use a delayed auto-shift system so holding a direction starts slow then speeds up, just like the original Tetris.
  • _UpdateFallTimer Hard — The heartbeat of Tetris. Each frame, the fall timer ticks up. When it reaches 30, the game checks whether the piece can move down. If it can, it drops one row. If it can't (something is underneath), a lock delay countdown begins. The player gets 30 frames of grace to slide or rotate before the piece locks permanently. When the lock expires, this function triggers the entire lock sequence: score the placement, copy tiles to the background, blink full rows, collapse cleared lines, and hide the sprites. This is the gravity, the tension, and the payoff — all in one function.

The Core Puzzle: Can This Piece Go Here?

The single most important function in the game answers one question: does a piece fit at a given position without overlapping anything?

  • _CanPieceBePlacedHere Hard — Walks through a piece's metasprite data (the list of 8×8 tile positions that make up the shape), checks each tile against the board boundaries and the background tilemap, and returns yes or no. Every other gameplay function depends on this: moving, rotating, falling, spawning, and detecting game over all call it.

Spawning and Placing Pieces

When the game starts (or after a piece locks), a new piece needs to appear at the top. When a piece reaches its final position, its tiles need to transfer from floating sprites into the static background:

  • _PickNewTetromino Medium — Takes the "next" piece that was shown in the preview, places it at the top of the board, and generates a new random "next" piece. If the new piece doesn't fit (the board is full), it returns failure — that's game over.
  • _SetCurrentPieceInBackground Hard — When a piece locks, this function reads the metasprite data and writes the piece's tiles and color attributes directly into the background tilemap. After this, the piece is part of the board — it's no longer a sprite and will be subject to row clearing.
  • move_sprite Easy
  • set_sprite_tile Easy
  • get_bkg_tile_xy Easy
  • set_bkg_tile_xy Easy
  • _RandomNumber Easy — Generates a random number from the Game Boy's hardware timer. Used to pick which piece comes next and what rotation it starts in. Simple but essential — without it, every game would be identical.

Clearing Lines: The Payoff

The heart of Tetris is clearing full rows. This requires four cooperating functions that scan the board, animate the clear, and collapse the remaining tiles:

  • _IsRowFull Medium — Checks whether all 10 columns in a given row contain non-blank tiles. Sounds simple, but it has a twist: during the blink animation, tiles alternate between being on the background and the window layer, so it needs a mode that checks both layers.
  • _BlinkFullRows Boss — The most complex function in the game. Scans for full rows, scores them, then plays an 8-cycle blink animation where full rows flash on and off. Uses the hidden window layer as a backup buffer so tiles can be restored after blanking. This is the function that gives the player the satisfying visual feedback of "I just cleared lines!"
  • _ShiftAllTilesAboveThisRowDown Hard — Clears one row and copies every row above it down by one position, handling both tile data and palette attributes. This is the gravity that fills the gap when a row disappears.
  • _ShiftAllTilesDown Medium — Scans from the bottom of the board upward. For each full row it finds, it calls the shift function above, then re-checks the same position (because a new full row may have dropped into its place). This handles clearing multiple rows at once.

The HUD: Keeping Score

The player needs to see their score, the current level, how many lines they've cleared, and a preview of the next piece. Four functions maintain this display:

  • _IncreaseScore Easy — Adds points to the 16-bit score variable and redraws the display. 5 points for placing a piece, 100 for each cleared line. A good introduction to 16-bit arithmetic on an 8-bit CPU (add the low byte, carry into the high byte).
  • _DrawNumber Hard — Converts a number to decimal digits and draws them as tiles on the background. Handles leading zeros and variable digit widths. Used to display the score (5 digits), level (2 digits), and line count (2 digits).
  • _UpdateGui Hard — Orchestrates the HUD: calls DrawNumber for each stat, then draws the next piece preview as a metasprite in the sidebar. Ties together the scoring system and the visual display.
  • _ResetAllSprites Easy — Moves all 40 hardware sprites offscreen and zeroes their tiles. Called during setup to ensure no ghost sprites linger from a previous round. A good first function to implement — it's a simple loop that teaches you how to call runtime functions and manage the stack.

The Main Loop Glue

Two small functions connect your game logic to the main loop's timing and piece lifecycle. They're both short but teach important concepts:

  • _UpdateFallTimer Hard
  • wait_vbl_done Easy — Pauses the CPU until the screen finishes drawing the current frame. This is how the game runs at exactly 60 frames per second — without this sync, the game would run as fast as the CPU allows, making it unplayable. Teaches the halt instruction and interrupt-driven programming in just 8 lines.
  • hide_metasprite Medium — When a piece locks, the main loop calls this to hide the piece's sprites (since the tiles are now part of the background). It walks through the metasprite entry list — the same list you traverse in _CanPieceBePlacedHere — but instead of checking collision, it moves each sprite to position (0, 0) which is offscreen. This is the moment the piece visually disappears as a sprite and "becomes" the board.

Tying It All Together

One final function initializes all the game state for each new round:

  • _UpdateFallTimer Hard
  • wait_vbl_done Easy
  • hide_metasprite Medium
  • _SetupGameplay Medium — Clears the board, clears the window layer, resets all sprites, picks the first piece, zeroes the score/lines/timer, sets the level to 1, and draws the initial HUD. This is the reset button — called once at the start and again after every game over.

The main loop in asm/main.asm (which is already written for you) calls these functions in the right order each frame: animate, handle input, fall, lock, clear, repeat. Your job is to make each piece work. When all 18 are implemented, you have Tetris.

5. move_sprite Easy

File: asm/runtime.asm

DirectionRegisters
InputA = sprite index (0-39), B = x position, C = y position
OutputNone (sprite position updated in shadow OAM)

Description

The Game Boy has 40 sprite slots. Each slot is 4 bytes in OAM (Object Attribute Memory): Y position, X position, tile number, and attribute flags. This function sets the Y and X bytes for one sprite.

The sprites aren't written to hardware OAM directly — they go into a "shadow OAM" buffer in regular RAM (wShadowOAM), which gets copied to the real hardware during VBlank by the DMA routine. This is the standard approach because hardware OAM is only accessible during specific timing windows.

To find the right sprite's bytes in the buffer, multiply the sprite index by 4 (since each sprite occupies 4 bytes). This is one of the first patterns you'll use over and over: index × entry_size = memory offset.

Your Task

This is the very first function you'll write. It's only about 8 instructions and teaches the fundamental pattern of computing a memory address from an index.

move_sprite::
    ; A = sprite index (0-39), B = x, C = y
    ;
    ; We need to find this sprite's slot in the shadow OAM buffer.
    ; Each sprite takes 4 bytes, so sprite N starts at wShadowOAM + N*4.
    ; The OAM layout for each sprite is: [Y] [X] [tile] [attributes]

    ; TODO: Compute the address of this sprite's OAM entry.
    ;   First, get the index into HL so we can do 16-bit math:
    ;     "ld h, 0" and "ld l, a" puts A (sprite index) into HL.
    ;   Multiply by 4: "add hl, hl" doubles HL (so twice = *4).
    ;   Add the base address of the shadow OAM buffer:
    ;     "ld de, wShadowOAM" then "add hl, de".
    ;   Now HL points to this sprite's Y byte.
    ;   (6 instructions)

    ; TODO: Write Y and X positions into the OAM slot.
    ;   The Y position is in C. Write it: "ld [hl], c".
    ;   Advance to the next byte: "inc hl".
    ;   The X position is in B. Write it: "ld [hl], b".
    ;   (3 instructions)

    ret

move_sprite::
    ld h, 0
    ld l, a
    add hl, hl
    add hl, hl                      ; HL = id * 4
    ld de, wShadowOAM
    add hl, de                      ; HL = &wShadowOAM[id]
    ld [hl], c                      ; Y position
    inc hl
    ld [hl], b                      ; X position
    ret

7. get_bkg_tile_xy Easy

File: asm/runtime.asm

DirectionRegisters
InputB = x (column 0-31), C = y (row 0-31)
OutputA = tile index at that position

Description

Reads one tile from the background tilemap. The background is a 32×32 grid of tile indices starting at VRAM address _SCRN0 ($9800). The address of tile (x, y) is _SCRN0 + y × 32 + x.

A helper function _BkgTileAddr is already written for you — it takes B=x, C=y and returns the VRAM address in HL. You also need to call _WaitVRAM before reading, because VRAM isn't always accessible (the video hardware locks it during screen drawing).

This function is used by _CanPieceBePlacedHere (to check if a cell is occupied) and _IsRowFull (to check if a row is completely filled).

Description

  • If (column XOR row) AND 1 is 0 → TILEANIMATION_TILE1_VRAM

A nested loop that scans the entire background. The XOR trick for the checkerboard pattern is a classic technique -- XOR two bits and you get 0 when they match, 1 when they differ.

13. _CanPieceBePlacedHere Hard

File: asm/Tetrominos.asm

DirectionRegisters
InputB = piece index (0-6), C = rotation (0-3), D = column, E = row
OutputA = 1 if piece fits, 0 if blocked or out of bounds

Description

The most critical collision detection function in the game. Determines whether a tetromino at the given position and rotation would overlap with existing tiles or fall outside the playing field. The function looks up the metasprite pointer from _Tetrominos_metasprites[piece*4 + rotation]. Each metasprite entry contains {dy, dx, dtile, props} and is terminated by a sentinel byte of $80. It walks through each sprite in the metasprite, accumulating position offsets: finalColumn += dx/8, finalRow += dy/8 (using arithmetic right shift 3 times for signed division by 8). For each tile position, it checks:

  1. If finalRow - 2 >= 18, the tile is below the board → return 0
  2. If finalRow < 2, the tile is above the board → skip bounds check (valid)
  3. If finalColumn >= 1 and finalRow >= 2, read the background tile at (finalColumn-1, finalRow-2) and compare with _blankTile. If not blank → return 0 (occupied)
The -1 and -2 offsets account for the sprite coordinate system where sprites are offset by 8 pixels horizontally and 16 pixels vertically from background coordinates.

Your Task

The most important function in the game. Everything else depends on it: moving, rotating, falling, spawning, and game-over detection. The metasprite walk pattern (read dy/dx, shift right by 3, accumulate position) appears again in SetCurrentPieceInBackground.

_CanPieceBePlacedHere::
    ; B=piece (0-6), C=rotation (0-3), D=column, E=row -> A=1 fits, A=0 blocked
    ;
    ; This is the heart of the game. Every time the player moves, rotates, or
    ; a piece falls, the game asks: "would the piece overlap anything here?"
    ;
    ; A piece is defined by its metasprite -- a list of {dy, dx, dtile, props}
    ; entries. We walk through each entry, accumulate the position offsets,
    ; and check whether that cell on the board is free.

    push de                         ; save the starting column (D) and row (E)

    ; ---- Look up which metasprite describes this piece+rotation ----
    ; There are 7 pieces x 4 rotations = 28 metasprites in the table.
    ; piece*4 + rotation gives us the index.
    ld a, b
    sla a                           ; double (piece*2)
    sla a                           ; double again (piece*4)
    add c                           ; add rotation

    ; Index into the pointer table to get the metasprite address
    ld c, a
    ld b, 0                         ; BC = table index
    ld hl, _Tetrominos_metasprites
    add hl, bc
    add hl, bc                      ; *2 because each pointer is 2 bytes
    ld a, [hli]                     ; read low byte of pointer
    ld c, a
    ld a, [hl]                      ; read high byte of pointer
    ld b, a                         ; BC now points to the metasprite data

    pop de                          ; restore D=column, E=row

    ; ---- Walk through each tile in the metasprite ----
    ; D = running column position, E = running row position
    ; BC = pointer to current metasprite entry
.loop:
    ; The first byte of each entry is dy (vertical offset in pixels).
    ; A value of $80 (128) is the "end of list" sentinel -- piece fits!
    ld a, [bc]
    cp $80                          ; end of metasprite?
    jr z, .pass
    inc bc                          ; advance past the dy byte

    ; TODO: Move our row position by dy/8.
    ;   A contains dy as a SIGNED pixel offset (e.g., 8 means "one tile down").
    ;   Divide by 8 using arithmetic shift right: "sra a" three times.
    ;   "sra" preserves the sign bit, so -8 correctly becomes -1.
    ;   Add the result to E (our running row): "add e", then "ld e, a".
    ;   (5 instructions)

    ; TODO: Read dx from [bc], advance bc, then move our column the same way.
    ;   Load [bc] into A, inc bc, then "sra a" three times, add D, store in D.
    ;   (6 instructions)

    ; Skip the dtile and props bytes -- we only need position for collision.
    inc bc                          ; skip dtile
    inc bc                          ; skip props

    ; TODO: Bounds check -- is this tile below the bottom of the board?
    ;   The board occupies rows 0-17 in "board coordinates" (after subtracting 2
    ;   for the sprite offset). So if (row - 2) >= 18, we're off the bottom.
    ;   But be careful: if row < 2, the subtraction would wrap around to a huge
    ;   number! Check "row < 2" first and skip the bounds check (tile is above
    ;   the visible board, which is fine).
    ;   (about 5 instructions: cp 2, jr c to skip, sub 2, cp 18, jr nc to .fail)

    ; TODO: Check if the cell at this position is occupied.
    ;   Only meaningful if column >= 1 AND row >= 2 (otherwise we're off the
    ;   edge of the play area, which is allowed).
    ;   If both conditions pass:
    ;     Save BC and DE (push them!) -- get_bkg_tile_xy destroys them.
    ;     Call get_bkg_tile_xy(B=column-1, C=row-2) to read the board.
    ;     Compare the result with [_blankTile].
    ;     Restore BC and DE (pop them!).
    ;     If the tile ISN'T blank, the cell is occupied -> jump to .fail.
    ;   (about 15 instructions including push/pop/call)

    jr .loop                        ; this tile is OK, check the next one

.pass:

    ret

    xor a                           ; at least one tile is blocked
    ret

23. wait_vbl_done Easy

File: asm/runtime.asm

DirectionRegisters
InputNone
OutputNone (returns after VBlank interrupt fires)

Description

This tiny function is how the game runs at exactly 60 frames per second. The Game Boy screen refreshes 60 times per second, and during each refresh there's a brief "VBlank" period when it's safe to update video memory. The VBlank interrupt handler (in header.asm, already written) sets wVBlankFlag to 1 each time VBlank occurs.

This function clears that flag to 0, then puts the CPU to sleep with halt. The CPU stays asleep (using zero power) until ANY interrupt fires. When it wakes up, it checks the flag — if it's still 0, a different interrupt woke us and we go back to sleep. When the flag is 1, VBlank happened and we return. This is called once per frame in the main loop.
